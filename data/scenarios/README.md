# Scenario extraction dataset

The JSONL files train and evaluate a compact model on a narrow task:
translating football narratives into typed semantic factors.

## Fields

- `id`: stable example identifier
- `home_team`, `away_team`: team-reference context
- `text`: scenario supplied to the extractor
- `expected`: target `ScenarioExtraction`
- `provenance`: how the record was produced
- `review_status`: `pending` or `approved`

The checked-in records are compositional synthetic fixtures with split-specific
language banks. They include multi-factor scenarios, negation, ambiguity,
irrelevant commentary, unsupported claims, and prompt-injection attempts.
They exercise generalization better than the original template placeholder,
but they are still not a claim-ready evaluation set. Every validation and test
record must be manually reviewed before competition metrics are published.
`scripts/evaluate_extractor.py` exposes a `claim_ready` flag and warning.

Do not train the final submitted adapter on the checked-in compositional
fixtures. Use them to test the data pipeline and audits. Generate the candidate
training corpus on Modal, review it, and retain a manually approved frozen test
set that was never used for prompt iteration or training.

Showcase match pairs are excluded from the generated dataset to reduce demo
contamination.

Run the leakage and coverage audit:

```bash
PYTHONPATH=src python scripts/audit_dataset.py
```

For higher-diversity candidate data generated by a large open model on Modal:

```bash
modal run training/modal_generate_dataset.py \
  --split train \
  --count 700 \
  --model-id Qwen/Qwen3-32B
```

Download the resulting JSONL from the `underdog-lab-training` Modal Volume,
validate it against the schema, and review it before replacing a split.

## Recommended review procedure

1. Review team attribution, factor class, severity, and certainty.
2. Add negation, contradiction, irrelevant-text, and prompt-injection cases.
3. Use two reviewers for the frozen test split.
4. Resolve disagreements before setting `review_status` to `approved`.
5. Never edit the test split after comparing base and tuned models.

Use the review helper:

```bash
PYTHONPATH=src python scripts/review_dataset.py data/scenarios/validation.jsonl
PYTHONPATH=src python scripts/review_dataset.py data/scenarios/test.jsonl
```

## Adjustment magnitudes (`src/underdog_lab/scenarios/adjustments.py`)

The `RULES` table's `affected_attack`/`opponent_attack`/`both_attacks`
multipliers (e.g. `-0.12` for `KEY_ATTACKER_UNAVAILABLE`) are hand-set
product assumptions, not fit from a historical injury/lineup dataset --
no such dataset exists in this repo, and none is referenced here without
one actually being sourced.

What *is* done: `adjustments.ruleset_metadata()` converts every magnitude
into an Elo-equivalent shift using the fitted model's own `elo_scale`
(`models/elo_fit_report.json`), via
`log(1 + multiplier_delta) / MODEL.elo_scale`. This grounds each
adjustment against a scale the model itself was fit on, rather than
leaving it as an unanchored percentage. Every rule's Elo-equivalent is
smaller in magnitude than the model's own fitted `home_advantage_elo`
(~66 points) -- the largest, `KEY_ATTACKER_UNAVAILABLE` at roughly -62
points, is still below it. So no hand-set scenario adjustment can swamp
a larger, data-fit effect.

If a historical lineup/absence dataset becomes available, the next step
is to fit these magnitudes directly (regress outcome residuals on absence
flags using the same MLE machinery as `fit_elo_dixon_coles.py`) and bump
`RULESET_VERSION`. Until then, `ruleset_metadata()`'s `limitation` field
should keep saying so.
